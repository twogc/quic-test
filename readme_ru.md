## О проекте

Этот репозиторий содержит комплексный набор инструментов и лабораторных работ для изучения современного протокола QUIC, разработанный в рамках научно-исследовательской деятельности Автономной некоммерческой организации "Центр исследований и разработок сетевых технологий" (далее — АНО CloudBridge Research).

Проект является частью инициативы АНО CloudBridge Research по развитию образовательных программ в области сетевых технологий и создание открытых образовательных материалов для вузов Российской Федерации.

**Интеграция с AI/ML:** Этот тестовый комплект тесно интегрирован с проектом [AI Routing Lab](https://github.com/twogc/ai-routing-lab), который использует метрики производительности QUIC для обучения моделей машинного обучения и прогнозирования оптимальных маршрутов в сетевой инфраструктуре CloudBridge. Подробнее см. раздел [Интеграция с машинным обучением (AI/ML)](#интеграция-с-машинным-обучением-aiml).

## Компоненты системы

### 1. Go Application (QUIC Tester)
Основное приложение на языке Go, реализующее:
- Тестирование протокола QUIC с различными параметрами
- Сбор метрик производительности в реальном времени
- Имитацию различных сетевых условий (задержки, потери пакетов, переупорядочение)
- Тестирование безопасности (анализ TLS 1.3, QUIC-специфичные проверки)
- Развертывание в облачной инфраструктуре российских провайдеров (Yandex Cloud, VK Cloud, Rostelecom и др.)

**Языки:** Go 1.25+
**Основные библиотеки:** quic-go, Prometheus

### 2. Rust Application (QUIC Bottom)
Визуализационное приложение на Rust, предоставляющее:
- Профессиональный пользовательский интерфейс (TUI) для реального времени
- Визуализацию показателей производительности (графики, тепловые карты)
- Интерактивные элементы управления для динамического изменения параметров
- Анализ аномалий и корреляцию между метриками
- Экспорт результатов для дальнейшего анализа

**Языки:** Rust 1.70+
**Основные библиотеки:** Ratatui, Warp, Tokio

### 3. HTTP API Bridge
Система взаимодействия между компонентами:
- Go приложение отправляет метрики на HTTP сервер (порт 8080)
- Rust приложение подписывается на обновления метрик
- Обмен данными в JSON формате для совместимости

## Сценарии использования

### Сценарий 1: Базовое изучение QUIC (Лабораторная работа)
```bash
# Запуск сервера с визуализацией
./bin/quic-test --mode=server --addr=:9000 --quic-bottom

# В отдельном терминале: запуск клиента
./bin/quic-test --mode=client --addr=127.0.0.1:9000 --quic-bottom
```
**Цель:** Понимание базовой архитектуры QUIC и процесса обмена данными.

### Сценарий 2: Тестирование производительности (Научное исследование)
```bash
# Тестирование с различными параметрами сети
go run main.go --mode=test --network-profile=mobile --duration=60s --quic-bottom

# Тестирование с Forward Error Correction
go run main.go --mode=experimental --fec=0.1 --cc=bbrv2 --quic-bottom
```
**Цель:** Сравнение производительности в различных условиях и отправка результатов для анализа.

### Сценарий 3: Исследование безопасности (Курсовая/дипломная работа)
```bash
# Полное тестирование безопасности
go run main.go --mode=test --security-test --attack-simulation --quic-bottom
```
**Цель:** Анализ уязвимостей и эффективности защитных механизмов QUIC.

### Сценарий 4: Облачное развертывание (Продвинутые исследования)
```bash
# Развертывание в Yandex Cloud с автомасштабированием
go run main.go --mode=test --cloud-deploy --provider=yandex --region=ru-central1 --quic-bottom
```
**Цель:** Оценка масштабируемости и производительности в реальной облачной инфраструктуре российских провайдеров.

### Сценарий 5: TCP-over-QUIC туннелирование (Исследовательская работа)
```bash
# Сравнение: чистый TCP (бенчмарк)
go run main.go --mode=test --protocol=tcp --network-profile=satellite --duration=60s --report=tcp_satellite.md

# TCP через QUIC туннель (CloudBridge архитектура)
go run main.go --mode=tcp-tunnel --network-profile=satellite --cc=bbrv3 --fec=0.1 --duration=60s --report=tcp_over_quic_satellite.md

# Сравнение с чистым QUIC
go run main.go --mode=test --protocol=quic --network-profile=satellite --duration=60s --report=quic_satellite.md
```
**Цель:** Исследование эффективности туннелирования TCP приложений поверх QUIC с оптимизациями BBRv3 и FEC в условиях нестабильных каналов (мобильные сети, спутниковая связь). Демонстрация преимуществ overlay-сети CloudBridge для Zero Trust архитектур.

## Интеграция с машинным обучением (AI/ML)

### AI Routing Lab - Прогнозирование оптимальных маршрутов

quic-test предоставляет метрики в реальном времени для обучения моделей машинного обучения и умного выбора маршрутов:

- **Экспорт метрик в реальном времени** - Prometheus метрики на порту 9090 с полной информацией о производительности QUIC
- **Автоматический сбор данных** - Сборщик метрик для latency, jitter, throughput и паттернов потерь пакетов
- **Обучение моделей** - Исторические данные метрик позволяют моделям предсказывать задержку и jitter на разных сетевых путях
- **Валидация моделей** - Обученные прогнозные модели валидируются на реальном QUIC трафике
- **Развертывание в производстве** - Оптимизированные модели интегрируются в CloudBridge Relay для умного выбора маршрутов

**Ключевые точки интеграции:**
1. **Экспорт метрик** - Go приложение экспортирует метрики Prometheus в реальном времени
2. **Сбор данных** - Сборщик AI Routing Lab получает метрики с экземпляров quic-test
3. **Прогнозные модели** - ML модели анализируют паттерны и предсказывают оптимальные маршруты
4. **Обратная связь** - Прогнозы валидируются на реальной сетевой производительности

**Пример быстрой интеграции:**

```bash
# Терминал 1: Запуск quic-test с экспортом Prometheus метрик
./bin/quic-test --mode=server --addr=:9000 --prometheus-port 9090

# Терминал 2: Сбор метрик с помощью AI Routing Lab
python -m data.collectors.quic_test_collector \
  --prometheus-url http://localhost:9090 \
  --output-file metrics.json

# Терминал 3: Обучение прогнозной модели
python -m models.train_predictor \
  --metrics-file metrics.json \
  --model-type lstm \
  --output-model model.pkl
```

**Подробнее:** См. [Руководство интеграции с AI Routing Lab](docs/integration/AI_ROUTING_LAB.md) для полной настройки и документации API.

**Связанные проекты:** [AI Routing Lab](https://github.com/twogc/ai-routing-lab), [CloudBridge Relay](https://github.com/twogc/cloudbridge-scalable-relay)

## Поддерживаемые протоколы и технологии

### Основные протоколы
- **QUIC (RFC 9000)** — основной протокол для изучения
- **HTTP/3 over QUIC** — приложение на базе QUIC
- **MASQUE** — туннелирование и проксирование
- **ICE/STUN/TURN** — NAT traversal и P2P соединения
- **TLS 1.3** — криптографическая основа QUIC

### Экспериментальные функции QUIC
- **BBRv2/BBRv3** — продвинутые алгоритмы управления перегрузкой
- **ACK Frequency** — оптимизация частоты подтверждений
- **Forward Error Correction (FEC)** — восстановление потерянных пакетов
- **Bit Greasing** — защита от будущей несовместимости

### TCP-over-QUIC (туннельный режим)
- **TCP-over-QUIC туннелирование** — моделирование overlay-сетей и Zero Trust туннелей
- **TCP приложения через QUIC** — тестирование реальных приложений (HTTP, SSH, RDP) внутри защищенного туннеля
- **Двойное управление перегрузкой** — исследование взаимодействия TCP и QUIC congestion control
- **Эффект FEC и BBRv3 на TCP-трафик** — анализ улучшений надежности при туннелировании

### Сетевые условия (имитация)
- **Excellent** — 5ms задержка, 0.1% потерь, 1 Gbps пропускная способность
- **Good** — 20ms задержка, 1% потерь, 100 Mbps
- **Poor** — 100ms задержка, 5% потерь, 10 Mbps
- **Mobile** — 200ms задержка, 10% потерь, 5 Mbps (с переупорядочением)
- **Satellite** — 500ms задержка, 2% потерь, 2 Mbps (с дублированием)
- **Adversarial** — 1000ms задержка, 20% потерь, 1 Mbps (с искажением)

## Организация лабораторных работ

### Типовая структура лабораторной работы

Каждая лабораторная работа включает:

1. **Цель работы** — четкое описание того, что студент должен изучить
2. **Теоретические основы** — краткое изложение необходимой теории
3. **Практические задания** — пошаговые инструкции для выполнения
4. **Контрольные вопросы** — проверка понимания материала
5. **Дополнительные задачи** — для продвинутых студентов
6. **Критерии оценивания** — как будет оцениваться работа

### Примеры рекомендуемых лабораторных работ

#### ЛР 1: "Основы протокола QUIC и организация взаимодействия клиент-серверной архитектуры" (3 часа)
- Запуск простого QUIC сервера и клиента
- Понимание структуры пакетов QUIC
- Анализ процесса handshake
- Измерение базовых метрик (latency, throughput)

#### ЛР 2: "Анализ производительности QUIC в различных сетевых условиях" (4 часа)
- Имитация различных сетевых условий (mobile, satellite, adversarial)
- Сравнение показателей производительности
- Анализ влияния задержки и потерь пакетов на пропускную способность
- Подготовка отчета с графиками

#### ЛР 3: "Экспериментальные функции QUIC: BBRv2 и Forward Error Correction" (5 часов)
- Тестирование алгоритма управления перегрузкой BBRv2
- Реализация Forward Error Correction
- Сравнение результатов с классическими алгоритмами
- Анализ trade-off между задержкой и надежностью

#### ЛР 4: "Тестирование безопасности QUIC" (4 часа)
- Анализ TLS 1.3 handshake в контексте QUIC
- Проверка валидности сертификатов
- Тестирование атак (MITM, replay, DoS)
- Документирование обнаруженных проблем

#### ЛР 5: "Облачное развертывание и масштабирование QUIC сервиса" (6 часов)
- Развертывание в облачной платформе Yandex Cloud, VK Cloud или Rostelecom Cloud
- Настройка load balancer
- Тестирование автомасштабирования
- Анализ затрат и оптимизация ресурсов

#### ЛР 6: "Сравнение TCP, QUIC и TCP-over-QUIC в нестабильных сетях" (8 часов, магистратура/исследовательская)
- **Цель:** Исследовать преимущества туннелирования TCP приложений поверх QUIC в условиях потерь, задержек и переменной пропускной способности
- **Части работы:**
  1. Тестирование чистого TCP в различных сетевых профилях (мобильная, спутниковая, враждебные условия)
  2. Тестирование чистого QUIC с BBRv3 и FEC в тех же условиях
  3. Тестирование TCP-over-QUIC туннеля (TCP приложение → QUIC туннель → TCP назначение)
  4. Сравнительный анализ трех подходов: throughput, latency, loss recovery, stability
  5. Исследование двойного congestion control (TCP + BBRv3 QUIC)
  6. Анализ эффекта FEC на TCP-трафик в туннеле
- **Ожидаемые результаты:** Количественные данные о преимуществах overlay-сетей для Zero Trust архитектур
- **Применение:** Реальная архитектура CloudBridge для защиты TCP приложений в движении

## Примеры команд для различных сценариев

### Базовое тестирование
```bash
# Запуск полного цикла (сервер + клиент)
go run main.go --mode=test --duration=20s

# Запуск отдельно сервера
go run main.go --mode=server --addr=:9000

# Запуск отдельно клиента
go run main.go --mode=client --addr=127.0.0.1:9000 \
    --connections=5 \
    --streams=10 \
    --packet-size=1200 \
    --rate=100
```

### Тестирование с визуализацией (требует отдельного терминала для QUIC Bottom)
```bash
# Терминал 1: запуск QUIC Bottom
./quic-bottom/target/release/quic-bottom-real

# Терминал 2: запуск теста с QUIC Bottom
./run_with_quic_bottom.sh --mode=test --duration=60s

# Просмотр интерактивного интерфейса в Терминале 1
```

### Имитация различных сетевых условий
```bash
# Мобильная сеть (высокая задержка, потери пакетов)
go run main.go --mode=test --network-profile=mobile --quic-bottom

# Спутниковый канал (очень высокая задержка)
go run main.go --mode=test --network-profile=satellite --quic-bottom

# Враждебные условия (высокая задержка и потери)
go run main.go --mode=test --network-profile=adversarial --quic-bottom
```

### Тестирование экспериментальных функций
```bash
# BBRv2 алгоритм управления перегрузкой
go run main.go --mode=experimental --cc=bbrv2 --quic-bottom

# Forward Error Correction с 10% избыточности
go run main.go --mode=experimental --fec=0.1 --quic-bottom

# Комбинация: BBRv2 + FEC + ACK Frequency
go run main.go --mode=experimental --cc=bbrv2 --fec=0.1 --ackfreq=5 --quic-bottom
```

### Тестирование безопасности
```bash
# Полное тестирование безопасности
go run main.go --mode=test --security-test --quic-bottom

# С имитацией атак
go run main.go --mode=test --security-test --attack-simulation --quic-bottom

# Анализ TLS версии и cipher suites
go run main.go --mode=test --security-test --tls-version=1.3 --quic-bottom
```

### Облачное развертывание (российские провайдеры)
```bash
# Yandex Cloud развертывание с автомасштабированием
go run main.go --mode=test --cloud-deploy --provider=yandex \
    --region=ru-central1 --instances=3 --quic-bottom

# VK Cloud развертывание
go run main.go --mode=test --cloud-deploy --provider=vk-cloud \
    --region=msk --instances=2 --quic-bottom

# Rostelecom Cloud развертывание
go run main.go --mode=test --cloud-deploy --provider=rostelecom \
    --region=moscow --instances=2 --quic-bottom
```

### TCP-over-QUIC туннелирование (исследовательское)
```bash
# Сценарий 1: TCP бенчмарк в спутниковом канале
go run main.go --mode=test --protocol=tcp \
    --network-profile=satellite \
    --duration=120s \
    --report=benchmark_tcp_satellite.json

# Сценарий 2: TCP через QUIC туннель (CloudBridge архитектура)
go run main.go --mode=tcp-tunnel \
    --network-profile=satellite \
    --cc=bbrv3 \
    --enable-fec \
    --fec-rate=0.1 \
    --duration=120s \
    --report=tcp_over_quic_satellite.json

# Сценарий 3: Сравнение с чистым QUIC
go run main.go --mode=test --protocol=quic \
    --network-profile=satellite \
    --cc=bbrv3 \
    --duration=120s \
    --report=quic_satellite.json

# Сценарий 4: Анализ двойного congestion control (TCP + QUIC)
go run main.go --mode=tcp-tunnel \
    --network-profile=mobile \
    --emulate-loss=0.15 \
    --cc=bbrv3 \
    --enable-fec \
    --monitor-congestion=both \
    --duration=120s \
    --report-format=json \
    --report=tcp_quic_cc_analysis.json

# Сценарий 5: Исследование эффекта FEC на TCP-трафик
go run main.go --mode=tcp-tunnel \
    --network-profile=loss-burst \
    --cc=bbrv3 \
    --fec-rate=0.05 --fec-rate=0.10 --fec-rate=0.15 \
    --duration=60s \
    --report=fec_impact_tcp.json
```

**Описание режимов:**
- `--protocol=tcp` — чистый TCP для бенчмарка
- `--protocol=quic` — чистый QUIC протокол
- `--mode=tcp-tunnel` — TCP туннелирование через QUIC (overlay-сеть)
- `--monitor-congestion=both` — мониторинг обоих congestion control алгоритмов
- `--fec-rate=X` — несколько значений избыточности FEC для сравнения
